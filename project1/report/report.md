# Normal-1 Performance Measurement (Search)

## Chapter 1. 实验目的

本实验的目标是实现并测试四种查找算法，分别为：

1. 二分查找迭代版
2. 二分查找递归版
3. 顺序查找迭代版
4. 顺序查找递归版

在此基础上，比较四种算法在最坏情况下的运行时间表现，并结合理论复杂度分析实验结果。

## Chapter 2. 程序设计与实现

### 2.1 问题描述

给定一个按升序排列的整数数组，共有 $N$ 个元素，元素值为 $0, 1, 2, \ldots, N-1$。本实验将待查找元素设置为 $N$，由于该元素不在数组中，因此可用于构造查找失败时的最坏情况。

实验要求比较两类查找方法：

1. 顺序查找：从左到右依次比较数组元素。
2. 二分查找：每次取中间位置，将查找区间缩小为原来的一半。

### 2.3 函数接口

四个核心函数的接口如下：

```c
int BinarySearchIterative(int arr[], int len, int x);
int BinarySearchRecursive(int arr[], int len, int x);
int SequentialSearchIterative(int arr[], int len, int x);
int SequentialSearchRecursive(int arr[], int len, int x);
```

其中：

1. arr 表示待查找的有序数组。
2. len 表示数组长度。
3. x 表示目标元素。
4. 若找到目标元素，返回其下标；否则返回 -1。

### 2.4 各算法实现思路

#### 2.4.1 二分查找迭代版

二分查找迭代版使用 left 和 right 两个变量表示当前查找区间。每轮循环计算中间位置 mid：

1. 若 arr[mid] == x，则返回 mid。
2. 若 arr[mid] < x，则说明目标仅可能出现在右半部分，令 left = mid + 1。
3. 若 arr[mid] > x，则说明目标仅可能出现在左半部分，令 right = mid - 1。

当循环结束后，若仍未匹配成功，则返回 -1。

#### 2.4.2 二分查找递归版

二分查找递归版每次比较当前区间的中间元素：

1. 若当前区间长度为 0，则说明查找失败，返回 -1。
2. 若中间元素等于目标值，则返回其相对位置。
3. 若中间元素大于目标值，则递归查找左半部分。
4. 若中间元素小于目标值，则递归查找右半部分，并在返回结果时补上偏移量。

与迭代版相比，该实现逻辑更直观，但会带来递归调用栈开销。

#### 2.4.3 顺序查找迭代版

顺序查找迭代版从数组下标 0 开始逐个比较：

1. 若 arr[i] == target，则立即返回下标 i。
2. 若扫描完整个数组仍未找到，则返回 -1。

由于最坏情况下需要检查所有元素，因此时间复杂度与数组长度成线性关系。

#### 2.4.4 顺序查找递归版

顺序查找递归版每次先检查当前数组首元素：

1. 若当前长度为 0，则返回 -1。
2. 若 arr[0] == x，则返回 0。
3. 否则递归地在剩余长度为 len - 1 的子数组中继续查找。
4. 若子问题查找成功，则在返回值基础上加 1，以恢复原数组中的正确下标。

该方法与迭代版在查找逻辑上等价，但递归实现会额外消耗栈空间。

### 2.5 测试程序设计

main.c 中完成了测试数据构造与计时：

1. 创建长度为 $N$ 的数组 arr。
2. 将数组初始化为 arr[i] = i，使其保持升序。
3. 将目标值设置为 target = N，从而保证查找失败，形成最坏情况。
4. 对每种算法重复执行 $K$ 次。
5. 使用 C 标准库中的 clock() 记录开始和结束时钟周期数。
6. 通过下式计算总运行时间：

$$
\text{Total Time} = \frac{\text{stop} - \text{start}}{\text{CLOCKS\_PER\_SEC}}
$$

再进一步得到单次调用平均耗时：

$$
\text{Duration} = \frac{\text{Total Time}}{K}
$$

为提高精度，应合理选择重复次数 $K$，使总 ticks 不小于 10。

## Chapter 3. 复杂度分析

### 3.1 最坏情况下的时间复杂度

| 算法 | 最坏情况说明 | 时间复杂度 |
| --- | --- | --- |
| 二分查找迭代版 | 每次排除一半区间，直到区间为空 | $O(\log N)$ |
| 二分查找递归版 | 每次递归缩小为原问题的一半 | $O(\log N)$ |
| 顺序查找迭代版 | 需要检查全部 $N$ 个元素 | $O(N)$ |
| 顺序查找递归版 | 递归检查全部 $N$ 个元素 | $O(N)$ |

### 3.2 空间复杂度分析

| 算法 | 空间复杂度 | 原因 |
| --- | --- | --- |
| 二分查找迭代版 | $O(1)$ | 只使用常数个辅助变量 |
| 二分查找递归版 | $O(\log N)$ | 递归深度约为 $\log N$ |
| 顺序查找迭代版 | $O(1)$ | 只使用循环变量 |
| 顺序查找递归版 | $O(N)$ | 最坏情况下递归深度为 $N$ |

从理论上看，两种二分查找在时间效率上优于两种顺序查找；而在同一算法思想下，迭代版通常比递归版具有更低的函数调用开销。

## Chapter 6. 结果分析

根据理论分析与实验设计，可以从以下几个方面对结果进行讨论：

1. 随着 $N$ 增大，二分查找的平均运行时间增长较慢，而顺序查找增长更明显。
2. 在相同查找思想下，迭代版通常会比递归版更快，因为递归调用存在额外函数开销。
3. 对于本实验中“查找失败”的最坏情况，顺序查找需要扫描所有元素，因此更容易体现线性增长趋势。
4. 二分查找虽然理论复杂度较低，但当 $N$ 较小时，常数项和计时误差可能使不同实现之间差距不明显。

可在填入测试数据后，结合折线图进一步分析以下问题：

1. 四条曲线是否与理论复杂度趋势一致。
2. 递归版与迭代版之间的性能差距是否稳定。
3. 当 $N$ 较小时，实验数据是否受到计时精度影响。

## Chapter 7. 结论

本实验完成了二分查找与顺序查找的迭代、递归共四种实现，并通过统一的测试程序对最坏情况下的运行时间进行了测量设计。理论分析表明：

1. 二分查找的最坏时间复杂度为 $O(\log N)$。
2. 顺序查找的最坏时间复杂度为 $O(N)$。
3. 迭代实现通常比递归实现具有更低的额外开销。

因此，随着数据规模增大，二分查找应明显优于顺序查找；而在相同算法下，迭代版一般略优于递归版。待补充完整实验数据与图表后，可进一步验证理论分析与实际测量结果的一致性。
